# Алгоритмы обработки контекстных правил в HOUND Semantic Engine

## Введение

Семантический движок HOUND разбирает SQL-скрипты и превращает их в граф знаний: вершины — это таблицы, колонки, процедуры, выражения; рёбра — это потоки данных, зависимости фильтрации, связи вызовов. Всё это происходит в один проход по дереву разбора (AST), с дополнительным постпроходным этапом для разрешения ссылок, которые были невозможны на этапе чтения.

Документ описывает алгоритмы управления контекстом — то, как движок знает, «где он сейчас находится» и что именно нужно делать с каждым встреченным идентификатором.

---

## 1. Стек контекстов — фундамент всего

### Назначение

В процессе обхода SQL-скрипта движок всегда находится внутри какого-то оператора: `SELECT`, `INSERT`, `PROCEDURE`, `CTE`, подзапроса и так далее. Операторы вкладываются друг в друга на произвольную глубину. Стек контекстов — это живая «стопка карточек»: каждая карточка описывает один активный оператор, самая верхняя карточка — это текущий активный оператор.

### Что хранит одна карточка (ScopeContext)

Карточка несёт:

- **GEOID** — иерархический строковый идентификатор оператора, например `PK_TEST:PROC1:INSERT:40` или `PK_TEST:PROC1:INSERT:40:SQ:C:42`.
- **Тип оператора** — `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `CTE`, `SUBQUERY`, `USUBQUERY`, `CURSOR`, `REF CURSOR`, `DINAMIC_CURSOR`, `VIEW` и другие.
- **GEOID родителя** — ссылка на карточку-предка для построения иерархии.
- **GEOID подпрограммы** — текущая процедура/функция/пакет.
- **Алиасы** — словарь «псевдоним → GEOID» для разрешения CTE и подзапросов внутри этого оператора.
- **Набор семантических флагов** — булевы переключатели, которые меняют смысл последующих атомов.

### Семантические флаги и приоритет их интерпретации

В любой момент активна ровно одна из следующих ситуаций. Они проверяются строго сверху вниз:

1. **Мы внутри VALUES-выражения** (`isInValuesClause`) — контекст атома становится `VALUES`.
2. **Мы в ON-условии JOIN** (`isInJoinContext`) — контекст атома становится `JOIN`.
3. **Мы в правой части UPDATE SET col = …** (`isInUpdateSetExpr`) — контекст становится `UPDATE_EXPR`.
4. **Мы в INSERT-части MERGE** (`isMergeInsertPart`) — контекст становится `INSERT`.
5. **Мы в UPDATE-части MERGE** (`isMergeUpdatePart`) — контекст становится `UPDATE`.
6. **Ничего из выше** — контекст равен типу текущего оператора: `SELECT`, `WHERE`, `HAVING`, `GROUP BY` и т.д.

Именно этот результат атом запоминает как `parent_context`. По нему позже писатель граф-базы решает, какой тип ребра создавать (DATA_FLOW, FILTER_FLOW и т.д.).

### Управление стеком

- **Вход в оператор** — новая карточка создаётся и кладётся на вершину стека. Все последующие атомы принадлежат этому оператору.
- **Выход из оператора** — верхняя карточка снимается. Управление возвращается родительскому оператору.
- Стек никогда не опустеет до нуля во время обхода скрипта, потому что скрипт сам по себе считается корневым контекстом.

## 2. Построение GEOID

### Зачем нужен GEOID

GEOID — это первичный ключ каждого оператора в графе. Он должен быть:
- **Уникальным** внутри одного скрипта
- **Содержательным** — по нему можно понять, что это за оператор и откуда он
- **Иерархическим** — вложенный оператор содержит GEOID родителя как префикс

### Правила построения

GEOID собирается из частей, разделённых двоеточием.

**Корневой компонент** — это GEOID родительского оператора или, если родителя нет, GEOID подпрограммы. Если и подпрограммы нет — GEOID пуст.

**Тип-компонент** зависит от вида вложенного оператора:

| Тип оператора | Компонент в GEOID |
|---|---|
| CTE | `CTE_ИМЯ` |
| Подзапрос в FROM с алиасом | `SQ:АЛИАС` |
| Подзапрос в FROM без алиаса | `SQ` |
| Скалярный подзапрос в UPDATE SET | `USQ:АЛИАС` или `USQ` |
| CURSOR / REF CURSOR | `ИМЯ_КУРСОРА` |
| Остальные DML | `ТИП_ОПЕРАТОРА` (INSERT, SELECT, UPDATE…) |

**Суффикс** — номер строки начала оператора в исходном файле. Это позволяет различать два `SELECT` на разных строках.

**Пример нарастания:**

```
PROC1                            ← процедура
PROC1:INSERT:40                  ← INSERT на строке 40
PROC1:INSERT:40:SQ:C:42          ← CTE "C" в составе INSERT
PROC1:INSERT:40:SQ:C:42:SQ:X:45 ← подзапрос внутри CTE
```

Если MERGE идёт вторым оператором в той же процедуре — он получает независимый корень `PROC1:MERGE:50`, а не `PROC1:INSERT:40:MERGE:50`. Именно это гарантирует корректность родительской цепочки.

## 3. Жизненный цикл оператора

### Фаза входа (onStatementEnter)

Когда парсер обнаруживает начало нового оператора:

1. Берётся текущее состояние стека: кто сейчас родитель, какая подпрограмма активна.
2. По этим данным строится GEOID нового оператора.
3. Создаётся карточка ScopeContext и кладётся на вершину стека.
4. Если у входящего оператора есть алиас (например, подзапрос `(SELECT ...) AS t`), алиас регистрируется в словаре **родительского** оператора — именно там он будет нужен для разрешения ссылок.
5. В строителе графа создаётся запись StatementInfo с ссылками на родителя и подпрограмму.

### Фаза выхода (onStatementExit)

Выход — самый насыщенный этап. Он выполняется в строгой последовательности:

**Шаг 1 — Раскрытие звёздочки.** Если оператор содержал `SELECT *`, движок пытается раскрыть его в явный список колонок. Для этого просматриваются source subqueries оператора: если там есть зарегистрированные output columns, они копируются в текущий оператор.

**Шаг 2 — Попытка разрешения атомов.** Все атомы текущего оператора обрабатываются NameResolver-ом. Те, что удалось разрешить, получают ссылку на DaliColumn. Неразрешённые записываются в список `pendingColumns` для второго прохода.

**Шаг 3 — Постобработка JOIN-ов.** Для каждого JOIN в операторе пробуется разрешить исходную и целевую таблицы, если они не были разрешены в момент регистрации.

**Шаг 4 — UNION-слияние.** Если оператор помечен как часть UNION, его output columns переносятся в родительский оператор (добавляются только уникальные имена).

**Шаг 5 — Регистрация как источника.** Если оператор — CTE, подзапрос, USUBQUERY или inline SELECT — он регистрируется в родительском операторе как source subquery. Это позволяет родителю ссылаться на него при разрешении атомов.

**Шаг 6 — VIEW-обработка.** Если оператор — это тело SELECT внутри CREATE VIEW, его output columns копируются на вершину-представление в графе.

**Шаг 7 — Курсорная пропагация (CURSOR-OC).** Курсоры не получают событий output column напрямую — их получает дочерний SELECT. Поэтому при выходе SELECT: если родитель является CURSOR, REF CURSOR или DINAMIC_CURSOR, output columns SELECT копируются в родительский оператор.

**Шаг 8 — INSERT-SELECT связывание (H1.2-INS).** Если текущий SELECT является дочерним оператором INSERT с явным списком колонок, каждый output column SELECT по позиции связывается с соответствующей целевой колонкой INSERT (через атрибут `table_ref`). INSERT получает подтип `INSERT_SELECT`.

**Шаг 9 — USUBQUERY связывание (H1.2-UPD).** Если текущий USUBQUERY (скалярный подзапрос в UPDATE SET) является дочерним UPDATE, его output columns связываются с соответствующими UPDATE-целевыми колонками. Берутся последние N записей из списка affected_columns типа UPDATE (где N = число output columns подзапроса).

**Шаг 10 — Снятие карточки.** Карточка снимается со стека. Управление возвращается родительскому оператору.

## 4. Атомы — единицы семантики

### Что такое атом

Атом — это минимальная осмысленная единица выражения в SQL. Каждый идентификатор, каждая константа, каждый вызов функции становится атомом. Атомы — это «сырой материал», из которого потом выстраивается граф зависимостей.

### Метаданные атома

Каждый атом несёт:

- **Текст** — исходный текст выражения (`emp.salary`, `MAX(amount)`, `42`).
- **Позиция** — строка и столбец начала и конца в файле.
- **parent_context** — контекст в момент создания (SELECT, WHERE, JOIN, UPDATE_EXPR и т.д.).
- **Признак сложности** — содержит ли атом точку или скобки.
- **Классификационные флаги** — является ли атом ссылкой на колонку, вызовом функции, константой.
- **Токены** — разбиение текста на лексемы, с типами токенов.
- **output_column_sequence** — номер позиции в SELECT-списке, если атом входит в output column.
- **dml_target_ref** — ссылка на колонку-цель (для атомов в UPDATE SET / MERGE UPDATE SET).

### Алгоритм классификации атома

После регистрации каждый атом проходит классификацию:

1. Если начинается с `:` — это bind-переменная. Константа.
2. Если состоит из одного токена-числа или строки — константа.
3. Если это псевдоколонка ROWNUM, SYSDATE, USER и подобные — константа.
4. Если текст содержит `(` — функциональный вызов.
5. Если состоит из пяти токенов вида `schema.table.column` — разбирается полный трёхчастный путь.
6. Если состоит из трёх токенов вида `table.column` — квалифицированная ссылка на колонку.
7. Если это одно слово — простая ссылка на колонку или переменную.
8. Особый случай: `sequence.NEXTVAL` / `sequence.CURRVAL` — ссылка на последовательность.

Классификация напрямую влияет на то, будет ли для этого атома создаваться ребро DATA_FLOW или FILTER_FLOW в граф-базе.

### Позиционная привязка к output column

После того как движок обработал один элемент SELECT-списка, он выполняет привязку: все атомы, чья позиция (строка:столбец) попадает в диапазон этого элемента, получают атрибут `output_column_sequence = N`, где N — порядковый номер колонки. Это позволяет знать, в какую именно output column «вытекает» данный атом.

## 5. SELECT — обработка списка вывода

### Регистрация output column

Когда парсер заканчивает обработку одного элемента SELECT-списка, движок:

1. Вычисляет порядковый номер: он равен текущему количеству уже зарегистрированных output columns плюс один.
2. Определяет имя колонки:
   - Если есть явный алиас (`col AS my_name`) — берётся алиас.
   - Иначе берётся выражение как есть.
   - Если вычислить невозможно — генерируется синтетическое имя `EXPR_N`.
3. Добавляет запись в упорядоченный словарь output columns оператора.
4. Вызывает позиционную привязку атомов к этому номеру.

### Звёздочка (SELECT *)

`SELECT *` регистрируется как единственный output column с именем `*` и типом `star`. Реальное раскрытие происходит позже, при выходе из оператора, когда известны все source tables и subqueries.

### UNION

Когда `UNION` обнаруживается между двумя SELECT-операторами:

- Каждый SELECT-оператор обрабатывается самостоятельно.
- При выходе из каждого части: его output columns переносятся в родительский оператор через механизм слияния.
- Слияние работает по принципу «добавить если не существует» — первый SELECT задаёт структуру.

## 6. Разрешение имён (NameResolver и pendingColumns)

### Когда происходит разрешение

Разрешение имён — это отображение «текстового имени таблицы или колонки» на конкретный GEOID в графе. Оно происходит на двух уровнях:

**Уровень 1 — При выходе из оператора.** Пока стек ещё содержит карточку оператора, движок пробует разрешить все атомы этого оператора. Здесь доступны алиасы и source tables текущего контекста.

**Уровень 2 — Пост-обход (resolvePendingColumns).** После завершения обхода всего AST движок делает финальный проход по списку неразрешённых атомов. Теперь известна полная картина всех таблиц и подзапросов в скрипте.

### Стратегии разрешения

NameResolver перебирает стратегии в порядке приоритета:

**1. Алиас-словарь.** Проверяется, не является ли имя зарегистрированным псевдонимом CTE или подзапроса. Если да — ссылка разрешается на соответствующий оператор, а не на таблицу.

**2. Source tables текущего оператора.** Проверяются все таблицы, зарегистрированные в FROM-части оператора. Квалифицированные ссылки (`table.column`) проверяются через конкретное совпадение имени или алиаса таблицы.

**3. Неявная таблица.** Если в операторе ровно одна source table, простые ссылки (`column`) разрешаются на неё без явного указания.

**4. Параметры и переменные подпрограммы.** Если атом — это простое имя, проверяется, не является ли оно параметром или переменной текущей процедуры/функции.

**5. Записи курсора.** Если атом вида `rec.field`, проверяется, не является ли `rec` переменной цикла курсора. Если да — поиск идёт через курсорный реестр в исходную таблицу курсора.

**6. Родительские контексты.** Для MERGE WHEN MATCHED / WHEN NOT MATCHED части движок просматривает родительские скоупы, чтобы найти source alias (`src.column`) или target alias (`t.column`).

**7. Текстовый фоллбэк.** Когда токены недоступны, разрешение выполняется по исходному тексту атома напрямую.

### Что происходит при успешном разрешении

- Атому присваивается `status = "Обработано"`.
- Атом получает ссылку на GEOID таблицы и имя колонки.
- В графе создаётся ребро ATOM_REF_COLUMN (атом → DaliColumn) и ATOM_REF_TABLE (атом → DaliTable).
- В лог записывается использованная стратегия разрешения.

### Что происходит при неудаче

- Атом остаётся в списке `pendingColumns`.
- После второго прохода, если атом всё равно не разрешён — он получает `status = "Не найден"` с причиной.
- В граф пишется DaliResolutionLog-запись для диагностики.

## 7. DML-контексты — обработка операторов изменения данных

### Общая логика DML

Для INSERT, UPDATE, DELETE, MERGE движок должен понять не только откуда берутся данные (source atoms), но и куда они записываются (target columns). Целевые колонки формируют список `affected_columns` — он потом становится вершинами DaliAffectedColumn в граф-базе.

### INSERT

**Явный список колонок** `INSERT INTO t (col1, col2) ...`:

Когда парсер прочитал список `(col1, col2)`, движок:
1. Регистрирует эти имена в `insertTargetColumns` оператора (упорядоченный список).
2. Для каждой колонки создаёт запись в `affected_columns` с типом `INSERT` и порядковым номером.

**VALUES-часть:**

При входе в VALUES-часть устанавливается флаг `isInValuesClause`, атомы получают контекст `VALUES`. После каждого выражения в VALUES счётчик позиции инкрементируется, атомы привязываются к позиции.

**INSERT SELECT:**

Если INSERT содержит не VALUES, а SELECT-подзапрос — это обнаруживается при выходе SELECT. Алгоритм H1.2-INS связывает output columns SELECT с целевыми колонками INSERT по позиции: первая output column → первая target column, и так далее.

### UPDATE

**Каждый элемент SET col = expr:**

1. Парсер сообщает движку имя целевой колонки (`col`).
2. Движок создаёт запись в `affected_columns` с типом `UPDATE`.
3. Эта запись выставляется как `column_affected` в текущем контексте.
4. Устанавливается флаг `isInUpdateSetExpr` — следующие атомы (из `expr`) получат `parent_context = UPDATE_EXPR`.
5. Атомы из выражения `expr` получают `dml_target_ref = GEOID_таблицы.COL` — ссылку на целевую колонку.
6. При выходе из clause флаг `isInUpdateSetExpr` снимается.

**Подзапрос в UPDATE SET** `SET col = (SELECT ...)`:

Подзапрос регистрируется как USUBQUERY. При его выходе алгоритм H1.2-UPD связывает output columns USUBQUERY с последними N записями UPDATE-типа в `affected_columns`.

**Многоколоночное присваивание** `(col1, col2) = (SELECT ...)`:

Все перечисленные колонки регистрируются как affected_columns подряд. Подзапрос привязывается по позиции.

### MERGE

MERGE — самый сложный DML. Он содержит source subquery, WHEN MATCHED и WHEN NOT MATCHED части.

**Source subquery USING (SELECT ...) AS src:**

Алиас `src` регистрируется в словаре **MERGE-оператора** (не подзапроса). Это позволяет в WHEN-частях резолвить `src.column` через алиас.

**WHEN MATCHED THEN UPDATE SET col = expr:**

При входе в каждый элемент SET:
1. Движок регистрирует целевую колонку как `MERGE_UPDATE_TARGET` в `affected_columns`.
2. Выставляет `column_affected` для последующих атомов.
3. Устанавливает `isInUpdateSetExpr`. Атомы в выражении получают `dml_target_ref`.
4. При выходе из элемента: `column_affected` и флаг сбрасываются.

**WHEN NOT MATCHED THEN INSERT (col1, col2) VALUES (...):**

При обнаружении списка `(col1, col2)`:
1. Каждая колонка регистрируется как `MERGE_INSERT_TARGET` в `affected_columns` с порядковым номером.

### DELETE

DELETE значительно проще. Атомы в WHERE получают контекст `WHERE`. Атомы в подзапросах FROM получают контекст `JOIN` или тип подзапроса. Специальных affected_columns для DELETE не регистрируется — тут нет целевых колонок.

## 8. JOIN-контекст

### Вход в JOIN

Когда парсер обнаруживает JOIN-клаузу, движок устанавливает `isInJoinContext = true`. Пока этот флаг активен, все атомы получают `parent_context = JOIN`. Это необходимо, чтобы атомы из ON-условия попадали в FILTER_FLOW, а не DATA_FLOW.

### Регистрация JOIN

Когда JOIN полностью разобран:
- Регистрируется JoinInfo: тип JOIN, GEOID исходной таблицы, GEOID целевой таблицы, псевдонимы, условие, номер строки.
- JoinInfo записывается в список joins текущего оператора.

### Выход из JOIN

Флаг `isInJoinContext` снимается. Атомы после JOIN-клаузы снова получают контекст из типа оператора или других флагов.

### Постобработка JOIN

При выходе из оператора для каждого JOIN делается попытка разрешить исходную и целевую таблицы, если это не удалось при регистрации. Используется тот же NameResolver.

## 9. CTE — Common Table Expressions

### Регистрация CTE

Когда парсер входит в CTE (`WITH name AS (SELECT ...)`):
1. Имя CTE пушится в стек псевдонимов (`subqueryAliasStack`).
2. Для CTE создаётся новый оператор типа `CTE` с GEOID вида `РОДИТЕЛЬ:ИМЯ_CTE:СТРОКА`.

### Использование CTE

При выходе из CTE:
1. Имя CTE поппится из стека.
2. Пара `имя → GEOID_CTE` регистрируется в словаре алиасов **родительского** оператора.

Когда позже в FROM-части встречается `FROM имя_CTE`:
- NameResolver находит его в словаре алиасов.
- Ссылка разрешается не как таблица, а как `source subquery` с GEOID CTE.
- В граф создаётся ребро `USES_SUBQUERY` от оператора к CTE.

### Вложенные CTE

CTE могут вкладываться — каждый регистрируется в словаре своего родителя. Важно, что CTE регистрирует алиас у **родительского** оператора, а не у самого себя — иначе он был бы недоступен для других CTE на том же уровне.

## 10. Курсоры и курсорные переменные

### Объявление курсора

При объявлении `CURSOR cur_name IS SELECT ...`:
1. Движок создаёт оператор типа `CURSOR` с GEOID `ПОДПРОГРАММА:ИМЯ_КУРСОРА:СТРОКА`.
2. Имя курсора регистрируется в курсорном реестре: `имя → GEOID`.
3. Дочерний SELECT курсора обрабатывается как обычно. При выходе SELECT алгоритм CURSOR-OC копирует его output columns в оператор CURSOR.

### REF CURSOR

`CURSOR` с динамическим запросом. Обрабатывается аналогично, но тип оператора — `REF CURSOR`. Алгоритм CURSOR-OC распространяется и на него.

### Цикл FOR с именованным курсором

`FOR rec IN cur_name LOOP`:
1. Переменная записи `rec` связывается с `cur_name` в курсорном реестре.
2. Атом вида `rec.FIELD` при разрешении: находим `cur_name` → находим GEOID курсора → находим source tables курсора → разрешаем `FIELD` в этих таблицах.

### Цикл FOR с inline-подзапросом

`FOR rec IN (SELECT ...) LOOP`:
1. Inline-подзапрос получает тип `DINAMIC_CURSOR`.
2. Переменная `rec` связывается с GEOID этого оператора.
3. Разрешение `rec.FIELD` аналогично: находим source tables DINAMIC_CURSOR → ищем FIELD.
4. Алгоритм CURSOR-OC также работает: дочерний SELECT копирует output columns в DINAMIC_CURSOR.

## 11. Inline-подзапросы в FROM (STAB-13)

### Проблема

`FROM (SELECT ...) AS t` — это подзапрос, выступающий источником данных. Его алиас `t` должен быть доступен для разрешения атомов в текущем операторе.

### Решение

1. При входе в подзапрос движку передаётся алиас (`t`).
2. Алиас регистрируется **в словаре родительского оператора** (не подзапроса).
3. GEOID подзапроса строится с учётом алиаса: `РОДИТЕЛЬ:SQ:T:СТРОКА`.
4. При выходе из подзапроса он регистрируется как source subquery родительского оператора с ключом `t`.
5. Теперь атомы вида `t.column` разрешаются через этот алиас на output columns подзапроса.

## 12. Правило dml_target_ref — цепочка от атома к цели

### Назначение

Когда колонка источника питает конкретную целевую колонку через UPDATE SET или MERGE UPDATE SET, это нужно зафиксировать в графе как ребро DATA_FLOW от DaliColumn к DaliAffectedColumn. Для этого каждый атом в выражении правой части должен «знать», в какую именно целевую колонку он идёт.

### Механизм

1. При регистрации целевой колонки (`onUpdateSetColumn` или `onMergeElementEnter`) в текущем контексте выставляется `column_affected` — ссылка на запись `affected_column` с полем `column_ref` (например, `DWH.T_ORDERS.AMOUNT`).
2. При каждом вызове `addAtom()`: если `column_affected` в контексте не пуст, атом получает поле `dml_target_ref = column_ref`.
3. При выходе из clause (UPDATE SET, MERGE-элемент): `column_affected` сбрасывается в `null`, флаг `isInUpdateSetExpr` снимается.

### Что происходит с dml_target_ref при записи в граф

Если атом разрешён (`status = "Обработано"`) и несёт `dml_target_ref`:
- В граф создаётся ребро DATA_FLOW от его DaliColumn-вершины к DaliAffectedColumn.
- На ребро навешивается `flow_type` в зависимости от типа оператора: `UPDATE`, `MERGE` или `INSERT`.

## 13. Типы рёбер потоков и как они выбираются

### DATA_FLOW (поток данных)

**Условие создания:** Атом разрешён, является ссылкой на колонку, имеет `output_column_sequence` (стоит в SELECT-списке) или `dml_target_ref` (стоит в правой части DML).

**Направление:** DaliColumn → DaliOutputColumn (для SELECT) или DaliColumn → DaliAffectedColumn (для DML).

**Атрибут flow_type:**

| Значение | Условие |
|---|---|
| `DIRECT` | Обычная ссылка в SELECT или DML-выражении |
| `AGGREGATE` | Оператор имеет агрегацию (`hasAggregation=true`) И атом находится в GROUP BY |
| `UPDATE` | Атом в UPDATE SET, oператор — UPDATE |
| `MERGE` | Атом в MERGE UPDATE SET или MERGE INSERT, оператор — MERGE |
| `INSERT` | Атом в INSERT, оператор — INSERT |

### FILTER_FLOW (поток фильтрации)

**Условие создания:** Атом разрешён, является ссылкой на колонку, находится в фильтрующей клаузе.

**Направление:** DaliColumn → DaliStatement (колонка участвует в фильтрации оператора).

**Атрибут filter_type:**

| Значение | Условие |
|---|---|
| `WHERE` | Атом в WHERE-части оператора |
| `HAVING` | Атом в HAVING-части |
| `JOIN` | Атом в ON-условии JOIN |

### JOIN_FLOW (поток соединения)

Создаётся между двумя таблицами, участвующими в JOIN. Направление: DaliTable → DaliTable через DaliJoin-вершину.

### UNION_FLOW (поток объединения)

Создаётся при UNION: связывает output columns частей UNION с output columns родительского оператора.

### ATOM_PRODUCES (атом → output column)

Вспомогательное ребро: каждый атом с `output_column_sequence` получает ребро ATOM_PRODUCES к своей DaliOutputColumn. Используется для точной трассировки: «из какого именно выражения вышла эта колонка».

## 14. Специальные конструкции

### CONNECT BY

Иерархический запрос. Атомы в `START WITH` и `CONNECT BY` получают контекст базового SELECT. Специальные псевдоколонки `LEVEL`, `CONNECT_BY_ROOT`, `SYS_CONNECT_BY_PATH` классифицируются как константы и не создают DATA_FLOW.

### PIVOT / UNPIVOT

При входе в PIVOT: список агрегатных функций и pivot-колонок регистрируются как output columns трансформированного оператора. Исходные строковые значения (IN-список) становятся именами новых output columns. Атомы в агрегатных функциях привязываются к соответствующим output columns.

### SELECT INTO (PL/SQL)

В PL/SQL `SELECT col INTO var FROM t` — атомы в `INTO var` не создают DATA_FLOW к output column таблицы. Переменная `var` регистрируется в контексте подпрограммы. Это частный случай INSERT-SELECT: output columns SELECT связываются с переменными по позиции.

### Пакетные операции (FORALL)

FORALL-блок обрабатывается как обёртка. Внутренний DML-оператор обрабатывается стандартно. Атомы коллекции (`collection(i)`) классифицируются как сложные выражения.

## 15. Диагностика и аудит

### ResolutionLog

Для каждого атома, независимо от результата разрешения, в `DaliResolutionLog` записывается:
- Исходный файл, таблица, колонка, позиция.
- Статус разрешения и использованная стратегия.
- Причина неудачи (если не разрешён).

Этот лог позволяет разработчику понять, почему конкретная колонка не была найдена: опечатка в имени, отсутствующая схема, нераскрытое *-выражение.

### SchemaLog

Если движок встречает имя схемы, которое подозрительно (не совпадает ни с одной известной) — оно записывается в `DaliSchemaLog` со стектрейсом.

### PerfStats

Для каждой сессии обработки замеряются и сохраняются временные метрики: время лексического разбора, синтаксического разбора, семантического обхода, записи в граф. Это позволяет выявлять медленные файлы.

## 16. Порядок алгоритмов в полном цикле обработки

Ниже — последовательность всех этапов от исходного SQL до заполненного графа:

1. **Лексический разбор** — SQL-текст разбивается на токены.
2. **Синтаксический разбор** — из токенов строится AST (дерево разбора).
3. **Семантический обход AST** — ParseTreeWalker обходит дерево, диалект-листенер вызывает события движка.
4. **Вход в подпрограмму** — создаётся ScopeContext подпрограммы, инициализируются её параметры и переменные.
5. **Вход в оператор** — создаётся ScopeContext, строится GEOID, регистрируются алиасы.
6. **Обработка FROM** — регистрируются source tables, JOIN-ы. Для каждого JOIN: флаг JOIN, атомы ON-условия, выход из JOIN.
7. **Обработка SELECT-списка** — каждый элемент порождает output column, атомы привязываются по позиции.
8. **Обработка WHERE/HAVING** — атомы получают соответствующий контекст.
9. **Обработка DML-целей** — INSERT-список, UPDATE SET-элементы, MERGE-части регистрируют affected_columns и выставляют флаги.
10. **Выход из оператора** — звёздочки раскрываются, атомы разрешаются (уровень 1), UNION/CTE/CURSOR-OC/H1.2 применяются.
11. **Выход из подпрограммы** — курсорный реестр закрывается, незакрытые скоупы принудительно закрываются с предупреждением.
12. **Пост-обход (resolvePendingColumns)** — второй проход по неразрешённым атомам.
13. **Запись в граф** — все собранные данные пишутся в ArcadeDB: вершины, рёбра потоков, диагностические документы.

## Приложение — словарь терминов

| Термин | Значение |
|---|---|
| **Атом** | Минимальная единица выражения: ссылка на колонку, константа, вызов функции |
| **GEOID** | Иерархический строковый идентификатор оператора |
| **ScopeContext** | Карточка активного оператора в стеке контекстов |
| **parent_context** | Тип SQL-клаузы, в которой находится атом (SELECT, WHERE, JOIN, UPDATE_EXPR…) |
| **affected_column** | Целевая колонка DML-оператора (INSERT target, UPDATE SET LHS, MERGE target) |
| **dml_target_ref** | Ссылка на affected_column, которую несёт атом из правой части DML-выражения |
| **output_column_sequence** | Порядковый номер output column, к которому привязан атом |
| **pendingColumns** | Список атомов, не разрешённых на уровне 1, ожидающих пост-обход |
| **poliage_update** | Метаданные DML-воздействия на колонку: type_affect (INSERT/UPDATE/MERGE) и order_affect |
| **CURSOR-OC** | Алгоритм копирования output columns из SELECT в родительский CURSOR-оператор |
| **H1.2-INS** | Алгоритм позиционного связывания SELECT output cols с INSERT target cols |
| **H1.2-UPD** | Алгоритм позиционного связывания USUBQUERY output cols с UPDATE target cols |
| **STAB-13** | Алгоритм обработки inline-подзапросов в FROM с алиасом |
| **flow_type** | Подтип DATA_FLOW ребра: DIRECT, AGGREGATE, INSERT, UPDATE, MERGE |
| **filter_type** | Подтип FILTER_FLOW ребра: WHERE, HAVING, JOIN |
